### `portfolio set-cash` safety — refuse destructive replace without `--confirm`
**Source:** Claude review (May 28, post-/pftui-report retrospective). pftui is a portfolio-management tool whose database stores real money positions for a single operator plus AI agents that call it via CLI.
**Why:** `pftui portfolio set-cash USD 173800` REPLACES every existing transaction for that currency with a single new row. The help text documents this ("replaces existing transactions for that currency"), but the destructive behavior is silent at runtime: no preview, no confirmation, no `--dry-run`. On May 28 a `set-cash` call collapsed 1 existing USD row; on a more-active book it could discard months of cash-movement history with zero warning. The command is also currently the only workaround for the unrelated paired-cash-leg bug (separate TODO above), so it is called more often than its design assumed — making the safety problem more urgent than the underlying choice suggests. An agent reading the help text learns the command "sets cash"; nothing tells them at the call site that they are deleting transactions.
**Scope:** (1) When `set-cash` would discard >1 existing transactions, require `--confirm`. If passed without it, exit non-zero with a printed summary of what would be discarded ("would discard N transactions totalling $X over date range Y..Z; pass `--confirm` to proceed"). (2) Always print a pre-action summary listing the IDs and dates of the rows about to be replaced (in `--json` mode, return it as a `discard_preview` array). (3) Add `--dry-run` that prints the preview and exits 0 without performing the replace. (4) Update the `--help` after_help text to lead with the destructive nature of the command. Files: `src/commands/portfolio.rs` (search for `set_cash` / `SetCash` handler), `src/cli.rs` for flag additions. Tests: `set-cash` refusing without `--confirm` when >1 rows would be discarded; `set-cash` allowing without `--confirm` when 0-1 rows; `--dry-run` not mutating DB; JSON output shape.
**Effort:** 1–2 hours.

### Allocation targets as ranges (floor + ceiling), not point values
**Source:** Claude review (May 28, post-/pftui-report retrospective).
**Why:** pftui's `allocation_targets` table stores a single `target_pct` plus a `drift_band_pct` (e.g., GC=F target 25% ± 2pp). The operating model the system actually serves is different: the operator is comfortable holding GC=F anywhere in the 22-30% band, with the band representing acceptable range based on conviction, not a tolerance around a single point. With the current schema, "drift" toward either edge of the band is treated as drift; in reality the operator has no preference between e.g. 23% and 27%, so neither should generate a "you should rebalance" signal. The current model also cannot express asymmetric tolerance ("happy down to 22%, anxious above 30%"). The fix is to replace `target_pct + drift_band_pct` with `target_floor_pct + target_ceiling_pct`, with drift only triggered when the actual position is outside the band on either side.
**Scope:** (1) Migration on `allocation_targets`: add `target_floor_pct TEXT`, `target_ceiling_pct TEXT`. Backfill from existing rows as `floor = target_pct - drift_band_pct`, `ceiling = target_pct + drift_band_pct`. Keep `target_pct` (legacy) for one release as a computed midpoint for backward compat, then drop in a follow-up. (2) `pftui portfolio target set <symbol> --floor <f> --ceiling <c>` becomes the new shape. The old `--target <t> --drift-band <b>` form still accepted (translated to floor/ceiling) for transitional compat. (3) `portfolio drift` reports drift relative to the nearer edge of the band, returning `0pp` if the position is inside the band. Output a new `band_position` enum: `below-floor | in-band | above-ceiling`. (4) `portfolio rebalance` only suggests trades when position is outside the band. (5) `portfolio brief` and the daily report consume the new schema. (6) AGENTS.md update documenting the new model. Files: `src/db/schema.rs` (migration), `src/db/allocation_targets.rs`, `src/commands/portfolio.rs`, `src/cli.rs`, `AGENTS.md`. Tests: in-band returns drift 0; floor edge returns `below-floor` and negative drift; ceiling edge `above-ceiling` and positive drift; backward-compat with `--target/--drift-band` produces equivalent floor/ceiling.

### Scenario probability math — explicit joint-probability model or renormalise to 100%
**Source:** Claude review (May 28, post-/pftui-report retrospective).
**Why:** Today's daily report contained the line "probabilities sum to 138% across the four — the scenarios are not mutually exclusive (Inflation Spike and Hard Recession can both occur as stagflation)." That line is honest but the math behind it is ambiguous to every downstream consumer. Are these four standalone probabilities (each is the marginal probability of that scenario, scenarios overlap)? Are they probabilities over a non-exhaustive sample space (the missing 38pp is "none of these")? Or are they independent draws (in which case stagflation = Inflation × Recession = 0.88 × 0.30 = 26%)? The current system stores `probability` as a single number per scenario, never specifying which interpretation applies. Agents reasoning over the scenarios will reach different conclusions depending on which interpretation they assume — bad for cross-run consistency, bad for any consumer wanting to plot scenario evolution.
**Scope:** Two viable paths, pick one with explicit decision recorded: (A) **Renormalise to 100% at the scenario-set level.** Probabilities are mutually exclusive over an exhaustive set; add an explicit "Other / None of these" residual scenario. Stagflation becomes its own scenario rather than a derived intersection. (B) **Explicit joint model.** Probabilities are marginal; add a new `scenario_joint_probabilities` table for explicitly-modeled intersections (e.g., `(scenario_a, scenario_b, joint_p)` for the Inflation × Recession = stagflation cell). The marginal sum is allowed to exceed 100%; consumers read the joint table when they need intersection probabilities. (1) Decision: pick A or B — A is simpler and is the default for most decision-theoretic systems; B is more expressive but requires more bookkeeping. Recommend A. (2) If A: migration to add an "Other / Unmodelled" scenario with `probability = 100 - sum(others)` and refuse to set scenario probabilities that sum >100. (3) If B: new table + CLI to set joints, daily report explicitly renders the marginal-vs-joint distinction. (4) Daily report rendering updated to match. (5) Document the choice in `docs/ANALYTICS-SPEC.md`. Files: `src/db/schema.rs` (migration), `src/db/scenarios.rs`, `src/commands/scenarios.rs`, `src/cli.rs`, `docs/ANALYTICS-SPEC.md`, the report skill. Tests: probability-sum constraint enforcement; daily-report rendering matches the chosen model.
**Effort:** 4–6 hours (path A) / 7–10 hours (path B).
