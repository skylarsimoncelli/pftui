### `pftui report` command family — scaffold native chart-rendering primitives (foundation)
**Source:** Skylar (May 28, follow-up to Claude review). Recognises that report visualizations currently live OUTSIDE pftui — in `~/pftui-operator/charts.py` (a separate Python module) and in ad-hoc Python scripts emitted by the Claude `/pftui-report` skill. Rendering depends on the skill orchestrator importing `charts.py` and writing markdown that embeds the returned SVG strings. The chart library is an orphan dependency.
**Why:** pftui's daily reports rely on visualizations (stacked bar of portfolio allocation, probability bars per scenario, regime quadrants, conviction-trajectory sparklines, decision cards, drift bars). These visualizations are the most operator-readable output the system produces. But they are not pftui features — they are claude-skill features that import a Python module that lives in a sibling directory. Consequences: (1) anyone using pftui outside the Claude skill cannot produce charts; (2) the data-to-chart pipeline crosses three runtimes (Rust pftui CLI → JSON → Python charts.py → SVG → markdown); (3) the chart library is untested by `cargo test`; (4) `pftui-operator/charts.py` is not versioned with pftui releases; (5) chart format / palette changes require coordination across two repos. The fix is to bring chart rendering into pftui as a first-class command family: `pftui report ...` produces SVG (or text-mode equivalents) natively in Rust, the data comes from the same SQLite the rest of the CLI reads, and the Claude skill (or any other consumer) just calls the command. This item is the FOUNDATION — it establishes the command tree, the SVG-rendering primitives, the palette/theme, and one or two reference chart implementations to validate the approach. Follow-on items port the rest of the helpers and wire the daily-report skill to use the native commands.
**Scope:** (1) Add a new top-level CLI namespace `pftui report` (alongside the canonical domains `agent`, `analytics`, `data`, `portfolio`, `system` — per CLAUDE.md's CLI Design Rules, `report` is a NEW canonical top-level domain and should be added to the canonical list in `CLAUDE.md`). (2) Subcommand structure: `pftui report chart <chart-name> [--from-db <query>] [--from-json <file>] [--out <file>] [--format svg|png|ascii]`. SVG is the primary format; PNG via the `resvg` crate; ASCII for terminal-friendly rendering. (3) Create `src/report/` module with submodules: `palette.rs` (color tokens — port from `pftui-operator/charts.py` PAL dict), `svg.rs` (SVG-string-building primitives: rect, line, text, group, with proper escaping), `theme.rs` (light/dark variants, sizing constants). (4) Implement TWO reference charts to validate the approach: (a) `pftui report chart stacked-bar` rendering portfolio allocation (data pulled from `portfolio status`) — port of `charts.py::stacked_bar`; (b) `pftui report chart prob-bar` rendering scenario probabilities with 7-day-prior ghost (data pulled from `scenarios` + `scenario_history`) — port of `charts.py::prob_bar`. (5) Both reference charts must produce SVG byte-identical (modulo whitespace) to the Python output for the same input — golden-file snapshot tests in `tests/report/snapshots/`. (6) Stand up the registry pattern: `src/report/registry.rs` mapping chart-name strings to render functions, so future charts plug in mechanically. (7) AGENTS.md update with a "Charts" section listing all available `pftui report chart` subcommands. (8) Read the existing implementations carefully to absorb the design — `~/pftui-operator/charts.py` is the source of truth for current chart semantics, the Claude skill at `~/.claude/skills/pftui-report.md` shows how they are consumed, the recent build script at `/tmp/build-private-report-2026-05-28.py` (may be deleted; reproduce by re-running the skill) shows data-to-chart wiring. Files: `src/report/mod.rs` (new), `src/report/palette.rs`, `src/report/svg.rs`, `src/report/theme.rs`, `src/report/registry.rs`, `src/report/charts/stacked_bar.rs`, `src/report/charts/prob_bar.rs`, `src/commands/report.rs` (new), `src/cli.rs`, `src/main.rs`, `CLAUDE.md` (add `report` to canonical domains), `AGENTS.md`, `tests/report/snapshots/` (new). ### Scenario probability math — explicit joint-probability model or renormalise to 100%
**Source:** Claude review (May 28, post-/pftui-report retrospective).
**Why:** Today's daily report contained the line "probabilities sum to 138% across the four — the scenarios are not mutually exclusive (Inflation Spike and Hard Recession can both occur as stagflation)." That line is honest but the math behind it is ambiguous to every downstream consumer. Are these four standalone probabilities (each is the marginal probability of that scenario, scenarios overlap)? Are they probabilities over a non-exhaustive sample space (the missing 38pp is "none of these")? Or are they independent draws (in which case stagflation = Inflation × Recession = 0.88 × 0.30 = 26%)? The current system stores `probability` as a single number per scenario, never specifying which interpretation applies. Agents reasoning over the scenarios will reach different conclusions depending on which interpretation they assume — bad for cross-run consistency, bad for any consumer wanting to plot scenario evolution.
**Scope:** Two viable paths, pick one with explicit decision recorded: (A) **Renormalise to 100% at the scenario-set level.** Probabilities are mutually exclusive over an exhaustive set; add an explicit "Other / None of these" residual scenario. Stagflation becomes its own scenario rather than a derived intersection. (B) **Explicit joint model.** Probabilities are marginal; add a new `scenario_joint_probabilities` table for explicitly-modeled intersections (e.g., `(scenario_a, scenario_b, joint_p)` for the Inflation × Recession = stagflation cell). The marginal sum is allowed to exceed 100%; consumers read the joint table when they need intersection probabilities. (1) Decision: pick A or B — A is simpler and is the default for most decision-theoretic systems; B is more expressive but requires more bookkeeping. Recommend A. (2) If A: migration to add an "Other / Unmodelled" scenario with `probability = 100 - sum(others)` and refuse to set scenario probabilities that sum >100. (3) If B: new table + CLI to set joints, daily report explicitly renders the marginal-vs-joint distinction. (4) Daily report rendering updated to match. (5) Document the choice in `docs/ANALYTICS-SPEC.md`. Files: `src/db/schema.rs` (migration), `src/db/scenarios.rs`, `src/commands/scenarios.rs`, `src/cli.rs`, `docs/ANALYTICS-SPEC.md`, the report skill. Tests: probability-sum constraint enforcement; daily-report rendering matches the chosen model.
**Effort:** 4–6 hours (path A) / 7–10 hours (path B).

### Adversary pseudo-analyst layer — argue against the convergence
**Source:** Claude review (May 28, post-/pftui-report retrospective).
**Why:** pftui's intelligence platform runs 4 timeframe analysts (LOW / MEDIUM / HIGH / MACRO) that produce "diverse" opinions per asset. In practice the four layers share priors: they read the same data bundle, the same lesson book, the same first-principles thesis context. They are more "the same lens at four focal lengths" than four independent lenses. When they appear to agree, the agreement may be confirmation of shared assumptions rather than independent corroboration. The system needs a structural counter-pressure: a fifth pseudo-layer whose explicit job is to argue against the current convergence using the same data, surface what each layer's assumptions exclude, and flag scenarios where consensus looks fragile. This is closer to a red-team than a fifth analyst. Today's report would have benefited: all four layers agreed today's hard-money capitulation is "positioning-driven, not structural." An adversary layer's job would be to write the strongest "actually, this IS structural" case using the same data, name the falsification triggers, and force the synthesis to address the counter-case explicitly.
**Scope:** (1) Create `agents/routines/adversary-analyst.md` — prompt template instructing the model to read the same bundles + analyst writes from the current run, identify the dominant convergence, and write the strongest opposing case using only data from those bundles. (2) Add a new author identifier `analyst-adversary` to the canonical list in `CLAUDE.md`. (3) The adversary runs AFTER the 4 timeframe analysts on each `/pftui-report` invocation (so it has their writes as input) but BEFORE synthesis. The adversary writes to a new table `adversary_views` with `(asset, current_convergence_summary, counter_case_summary, counter_case_evidence_points JSON, falsification_triggers JSON, fragility_score_1_5, recorded_at)`. (4) Synthesis MUST address the adversary's counter-case for any asset where `fragility_score >= 3`. (5) New CLI: `pftui analytics adversary --asset <SYM> --json`, `pftui analytics adversary fragility-rank --json`. (6) Daily report adds an "Adversary view" sub-section per asset where the fragility score is high — quoted directly from the adversary's write, not paraphrased. (7) Skill update: the report skill spawns the adversary subagent as a 5th parallel call OR sequentially after the 4 layers. Files: new `agents/routines/adversary-analyst.md`, `src/db/schema.rs` (migration), `src/db/adversary_views.rs` (new), `src/commands/analytics.rs`, `src/cli.rs`, `CLAUDE.md`, the report skill. Tests: adversary write/read; synthesis rejects publishing a report where any `fragility_score >= 3` view lacks a counter-case address in the markdown.
**Effort:** 2–3 weeks (substantial — touches the analytical pipeline core).

### MACRO analyst — falsifiable shorter-horizon checkpoints
**Source:** Claude review (May 28, post-/pftui-report retrospective).
**Why:** pftui's MACRO timeframe analyst (years-decades horizon) has 20 currently-open predictions and ZERO scored predictions in the trailing 60 days. By design — macro predictions resolve slowly. But the consequence is that MACRO is effectively uncalibrated: it cannot be wrong on any timescale that produces feedback, so its convictions never get refined by ground truth. Today MACRO held strong views on de-dollarisation, Stage 6 currency debasement, and Fourth Turning crisis-climax — none of which can be falsified on a horizon shorter than years. This is the same epistemic risk identified in the system's blind-spots register ("Geopolitical binary event underpricing" / "magnitude over-prediction" patterns — both were derived from layers that COULD be scored). The fix is to require MACRO to write quarterly checkpoint sub-predictions for each multi-year thesis: leading-indicator metrics that, if absent on a near horizon, would invalidate the macro call.
**Scope:** (1) Update `agents/routines/macro-timeframe-analyst.md`: for every active macro thesis (Stage 6, Fourth Turning, de-dollarisation, Dalio composite, structural inflation), the analyst MUST produce 2-3 quarterly checkpoint predictions on a 90-day horizon. Format: "By 2026-09-28, IF [observable leading indicator] is NOT [specific threshold], my [thesis X] is degraded." (2) Each checkpoint becomes a normal `user_prediction` row with `timeframe='macro-checkpoint'` (new enum value) and `target_date = recorded_at + 90 days`. (3) Existing macro `timeframe='macro'` predictions stay as multi-year structural calls (uncalibrated long-horizon by design). (4) Calibration display (separate TODO above) gets a `macro-checkpoint` row that ACTUALLY accumulates samples over time. (5) When a macro-checkpoint is scored Wrong, it triggers automatic re-evaluation of the parent thesis — surface in synthesis: "Macro thesis [X] has 1 of 3 checkpoints failed; analyst should re-examine before next run." (6) AGENTS.md update documenting the new pattern. Files: `agents/routines/macro-timeframe-analyst.md`, `src/db/schema.rs` (enum value), `src/db/user_predictions.rs`, `src/commands/prediction.rs`, the report skill (for the re-evaluation surface), `AGENTS.md`.
